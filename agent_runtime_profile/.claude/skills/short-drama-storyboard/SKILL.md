# 剧本改分镜

把长文本、剧本草案或已确认改编剧本拆成适合短视频平台传播的漫剧/短剧分镜脚本。

## 最高优先级边界

本 skill 是创作型分镜工具，用于用户主动要求“改成分镜脚本”时：

- 必须两步走：先分镜拆分大纲，用户确认后再出完整分镜。
- 默认输出到 `proposals/`，不直接覆盖 `scripts/episode_N.json`。
- 如用户确认要写入正式项目，必须通过 ArcReel 受控工具或主流程应用，不得 Bash/Write/Edit 直改正式项目数据。
- 本 skill 不生成图片、不生成视频、不写最终视频提示词。

## 输入优先级

- 如果输入是已确认剧本、调整后剧本、用户明确说已经改好的剧本：只负责拆集和拆分镜，不得再次重构剧情，不得改写人物动机，不得压缩或合并原剧情，不得新增剧情。
- 如果输入是小说、网文片段、故事梗概、剧本初稿，且用户明确要求“适配短视频节奏/加爆点/压缩/重构”：可以提出轻度结构调整建议，但仍需等待用户确认。
- 如果不确定输入是否已经确认，默认按“已确认剧本”处理，只拆分，不二次改写。

## script_policy 约束

开始前读取 `project.json` 的 `script_policy.mode`：

| 模式 | 行为 |
| --- | --- |
| `preserve` | 只能输出分镜草案到 `proposals/`，不得覆盖正式剧本。 |
| `suggest_rewrite` | 可以输出拆分大纲和分镜草案到 `proposals/`，用户确认后再应用。 |
| `rewrite_approved` | 用户确认后可由主流程把分镜草案应用到正式项目。 |

## 强制流程：必须分两步

绝对不要一步到位直接输出完整分镜。

### 第一步：输出分镜拆分大纲

收到用户剧本后，先做分镜拆分规划，只输出以下内容：

1. **剧本审查**
   - 核心卖点是什么
   - 人设卖点是什么
   - 哪些内容适合做每集开头爆点
   - 哪些内容适合做每集结尾钩子
   - 如果输入是已确认剧本，不提出删、挪、前置、压缩，只说明如何在不改剧情的情况下拆分

2. **整剧拆集大纲**
   按每集 1 分 30 秒 - 2 分 30 秒的体量拆分，输出：
   - 每集标题
   - 每集开头爆点
   - 每集核心剧情
   - 每集结尾钩子
   - 对应原剧本段落/幕/场次范围

3. **拆分说明**
   - 明确每集从原剧本哪里开始、哪里结束
   - 明确是否保持原剧本顺序
   - 明确没有二次重构剧情
   - 如确需调整顺序，单独列为“需用户确认的改动”，不得默认执行

最后补一句：

**“如果这版拆分大纲你确认，我再按这版继续生成完整分镜脚本。”**

然后停下，等待用户确认。

### 第二步：用户确认后输出完整分镜脚本

只有在用户明确确认分镜拆分大纲后，才可以生成完整分镜脚本。

## 分镜输出规则

每个分镜只包含以下固定字段：

1. **画面内容**
2. **对话/旁白**

不要在本阶段输出视频提示词、运镜参数、景别参数、光影参数。这些交给下游 `premium-video-prompt`。

## 分镜写法要求

- 分镜服务短视频节奏，不写成长篇文学描写。
- 每个分镜撑 8-15 秒画面量：把太碎的镜头合并，每个分镜包含 2-4 组画面动作和必要对白，确保生成视频时长够用。
- 一镜只承载一个核心信息点。
- 每镜明确“这段画面要让观众看到什么”。
- 开头几镜必须抓人，优先抛爆点、反差、冲突、悬念。
- 画面内容和对话/旁白尽量使用原文，不随意概括、扩写或改动人物动机。
- 如果原剧本已经是调整后的最终剧本，分镜不得再改写、扩写、重排、压缩剧情，只能拆成镜头块。
- 每一集必须形成：开头爆点、中段推进、结尾钩子。

## 平台适配原则

- 前 3 秒抓人。
- 人物关系尽快清楚。
- 已确认剧本保持原顺序；未确认初稿且用户明确要求改编时，冲突可适当前置。
- 信息表达要直，不绕。
- 不要过长铺垫。
- 每集结尾必须让人想点下一集。
- 画面比例读取项目配置，不硬编码竖屏。

## 改编边界

仅当输入不是已确认剧本，且用户明确要求改编时，才允许对原文做适度改编，但必须保持：

- 主线不变
- 人设核心不变
- 关键情绪不变

可以做：

- 对未确认初稿前置爆点
- 对未确认初稿使用倒叙开场
- 对未确认初稿压缩重复信息
- 对未确认初稿合并场景
- 对未确认初稿提前埋钩子
- 对未确认初稿把长段心理描写改成动作和冲突

不要做：

- 对已确认/已调整剧本进行二次重构
- 把拆分镜变成重新改剧本
- 在完整分镜阶段新增原剧本没有的剧情桥段
- 无理由魔改人物动机
- 把复杂剧情改到逻辑断裂
- 跳过大纲确认直接生成全部分镜

## 上下游关系

上游对应 `rewrite-script`。如果用户提供的是原始剧本且要求先优化，先用上游输出改编剧本草案，再将确认后的草案作为本 skill 输入。

下游对应 `premium-video-prompt`。本 skill 输出的分镜草案可作为下游输入；下游会按用户的视频提示词方法论生成可写回 ArcReel 前端红框的提示词。
